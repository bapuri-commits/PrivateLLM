## 1. 핵심 개념 정리

이 프로젝트에서 반복적으로 등장하는 용어와 개념을 먼저 정리합니다.

### LLM (Large Language Model)이란?

대규모 텍스트 데이터로 학습된 언어 모델. ChatGPT, Claude 같은 서비스의 핵심 엔진.
이 프로젝트에서는 **오픈소스 LLM**을 직접 다운로드받아 자체 GPU에서 실행합니다.

### "Uncensored" / "Abliterated"란?

- **Censored 모델**: OpenAI, Anthropic 등이 배포하는 모델은 안전성을 위해 특정 주제에 대해 "거절(Refusal)" 응답을 하도록 학습됨
- **Uncensored 모델**: 이 거절 메커니즘을 제거한 모델
- **Abliteration**: 모델의 내부 가중치에서 "거절을 담당하는 방향(direction)"을 수학적으로 찾아 제거하는 기법. 재학습 없이 기존 모델을 변환할 수 있음

### 양자화 (Quantization)란?

모델의 가중치를 더 적은 비트로 표현하여 **메모리 사용량을 줄이는 기법**.

| 형식 | 설명 |
|------|------|
| **FP16** (16비트) | 원본 품질. VRAM을 많이 차지 (70B 모델 ≈ 140GB) |
| **Q4** (4비트) | 원본 대비 75% 메모리 절감. 품질 손실 미미 (70B ≈ 35~40GB) |
| **GGUF** | llama.cpp 호환 양자화 포맷. CPU+GPU 혼합 실행 가능 |
| **EXL2** | ExLlamaV2 전용 고효율 포맷. GPU 전용, 약간 더 빠름 |

**핵심**: 양자화 덕분에 70B 파라미터 모델도 소비자용 GPU(48GB)에서 실행 가능합니다.

### VRAM이란?

GPU에 탑재된 전용 메모리. 모델 가중치 + 추론 중 생성되는 KV 캐시(대화 기억)가 여기에 올라갑니다.

- **모델 로딩**: Q4 양자화 기준, 파라미터 수(B) × ~0.5GB (예: 70B × 0.5 = 35GB)
- **KV 캐시**: 대화가 길어질수록(컨텍스트가 길수록) 추가 VRAM 소모
- **OOM (Out of Memory)**: VRAM이 부족하면 프로그램이 크래시남

### 파인튜닝 (Fine-Tuning)이란?

이미 학습된 모델을 **추가 데이터로 더 학습**시켜 특정 용도에 맞게 조정하는 것.

| 방식 | 설명 | VRAM 필요량 |
|------|------|------------|
| **Full Fine-Tune** | 모든 가중치를 업데이트 | 매우 큼 (70B → 수백 GB) |
| **LoRA** | 원본은 고정, 작은 "어댑터" 레이어만 학습 | 크게 절감 |
| **QLoRA** | LoRA + 4비트 양자화를 결합 | **24GB GPU로 70B 학습 가능** |

### LoRA / QLoRA 동작 원리 (좀 더 자세히)

1. 원본 모델 가중치는 **동결(freeze)** — 수정하지 않음
2. 각 레이어에 작은 저랭크(low-rank) 행렬 쌍 (A, B)을 추가
3. 학습 시 이 작은 행렬만 업데이트 → 원본 대비 **0.1~1% 수준의 파라미터만 학습**
4. QLoRA는 여기에 원본 가중치를 4비트로 양자화하여 VRAM을 추가 절감
5. 학습 완료 후: LoRA 어댑터를 원본에 **Merge**하여 단일 모델로 만들 수 있음

## 2. 모델 라인업 (Model Selection)

> 2026년 3월 기준으로 업데이트된 라인업입니다.

### 1차 타겟 (8B급 — 파이프라인 검증용)

파이프라인 전체(다운로드 → 양자화 → 추론 → 파인튜닝 → 머지 → 서빙)를 먼저 8B 모델로 검증합니다.
8B 모델은 RTX 4090(24GB)에서 매우 여유롭게 돌아가므로 실험 비용이 낮습니다.

| 모델 | HuggingFace 검색어 | 특징 | Q4 기준 VRAM |
|------|-------------------|------|-------------|
| **Llama 3 8B Abliterated** | `failspy/Meta-Llama-3-8B-Instruct-abliterated-v3` | 거절 메커니즘 수학적 제거. 가장 범용적 | ~5GB |
| **Dolphin Llama 3 8B** | `cognitivecomputations/dolphin-2.9.4-llama3-8b` | "무조건 복종" 데이터 학습. 시스템 프롬프트에 민감 | ~5GB |

### 2차 타겟 (70B급 — 품질 극대화)

8B에서 파이프라인이 검증되면, 큰 모델로 스케일업합니다.
70B 모델은 Q4 양자화 시에도 35~40GB가 필요하므로 **A6000(48GB) 이상 필수**입니다.

| 모델 | HuggingFace 검색어 | 특징 | Q4 기준 VRAM |
|------|-------------------|------|-------------|
| **Llama 3.3 70B Abliterated** | `huihui-ai/Llama-3.3-70B-Instruct-abliterated` | Llama 3.3 기반, 3.1 405B급 성능. 범용 최강 | ~35GB |
| **Midnight-Miqu 70B v1.5** | `sophosympatheia/Midnight-Miqu-70B-v1.5` | RP/창작 특화. 압도적 문장력, 32K 컨텍스트 | ~35GB |

### 모델 선정 시 고려사항

- **Abliterated 계열**: 기존 잘 학습된 Instruct 모델에서 거절만 제거. 지능 손실 최소
- **Dolphin 계열**: 처음부터 uncensored 데이터로 학습. 시스템 프롬프트 순종성이 높음
- **Mixtral 8x7B**: 원래 프롬프트에 있었으나, 2023년 모델이라 Llama 3.3 대비 세대가 오래됨. 필요 시 재검토
- 최종 모델은 **실제 테스트 결과**를 보고 결정 (Phase 1에서 비교)

## 3. 파인튜닝 데이터셋 (Training Data)

### 데이터셋이란?

모델에게 "이렇게 대답해라"라고 가르치기 위한 **예시 대화/텍스트 모음**.
보통 JSONL(JSON Lines) 형식으로, 한 줄이 하나의 학습 샘플입니다.

```jsonl
{"instruction": "사용자 질문", "input": "", "output": "모델이 해야 할 답변"}
```

### 검열 제거 및 순종성 강화 데이터

| 데이터셋 | 출처 | 설명 |
|----------|------|------|
| **Dolphin** | `cognitivecomputations/dolphin` | AI가 도덕적 훈계를 하지 않고 사용자 지시를 수행하도록 훈련된 방대한 데이터 |
| **OpenHermes 2.5** | `teknium/OpenHermes-2.5` | 논리, 코드, 창의적 글쓰기 등 지능 자체를 높여주는 100만+ 고품질 대화 |

### RP/묘사 특화 데이터

| 데이터셋 | 출처 | 설명 |
|----------|------|------|
| **PIPPA** | `PygmalionAI/PIPPA` | 실제 인간 대화 + 캐릭터 설정 기반. 대화 자연스러움 극대화 |
| **Virtuoso / LimaRP** | 커뮤니티 배포 | 소설적 문체, 감각적 상황 묘사 특화. 단조로운 답변 방지 |

### 데이터 전처리 (중요!)

원본 데이터를 그대로 쓰면 안 됩니다. 반드시 전처리가 필요합니다:

1. **포맷 변환**: 각 데이터셋의 고유 형식 → 학습 도구가 요구하는 JSONL/ChatML 형식으로 통일
2. **중복 제거**: 동일하거나 거의 같은 샘플 제거 (학습 효율 저하 방지)
3. **품질 필터링**: 너무 짧거나, 의미 없거나, 깨진 텍스트 제거
4. **토큰 길이 분포 확인**: 너무 긴 샘플은 VRAM을 과도하게 소모하므로 잘라내거나 제외
5. **비율 조정**: 각 데이터셋을 어떤 비율로 섞을지 결정 (예: Dolphin 60% + OpenHermes 30% + PIPPA 10%)

> **참고**: 실제로 데이터 전처리가 전체 작업 시간의 50% 이상을 차지하는 경우가 많습니다.

## 4. RunPod 인프라 구축 (Infrastructure)

### RunPod란?

GPU 서버를 시간 단위로 빌려 쓸 수 있는 클라우드 서비스.
자체 GPU를 구매하지 않아도 RTX 4090, A6000 등을 원격으로 사용할 수 있습니다.

### 하드웨어 구성

| 용도 | GPU | VRAM | RunPod 가격 (시간당) | 비고 |
|------|-----|------|---------------------|------|
| **파이프라인 검증 (8B)** | RTX 4090 | 24GB | ~$0.59 (Community) | 8B 모델 추론+파인튜닝 모두 가능 |
| **본격 운용 (70B)** | A6000 | 48GB | ~$0.49 (Community) | 70B Q4 모델 추론 가능. 파인튜닝은 QLoRA 필수 |
| **Storage** | Network Volume | 100GB+ | 별도 과금 | 모델 파일이 크므로 (70B ≈ 40~140GB) 충분한 공간 필요 |

### 비용 감각

| 작업 | 예상 소요 시간 | 예상 비용 |
|------|---------------|----------|
| 8B 모델 추론 테스트 | 2~4시간 | $1~3 |
| 8B QLoRA 파인튜닝 1회 | 4~8시간 | $3~5 |
| 70B 모델 추론 테스트 | 2~4시간 | $1~2 |
| 70B QLoRA 파인튜닝 1회 | 8~24시간 | $4~12 |

> **팁**: Pod을 사용하지 않을 때는 **Stop**하면 GPU 과금이 멈춥니다 (Storage 비용만 유지).

### 소프트웨어 스택

| 구성요소 | 기술 | 역할 |
|---------|------|------|
| **OS/런타임** | Docker `runpod/pytorch:2.2.1-py3.10-cuda12.1.1-devel-ubuntu22.04` | CUDA, PyTorch가 미리 설치된 환경 |
| **파인튜닝 도구** | **Unsloth** | QLoRA 학습을 2~5배 빠르게 실행. HuggingFace Transformers 드롭인 대체 |
| **추론 엔진** | **vLLM** | 고성능 추론 서버. OpenAI-compatible API 제공 |
| **실험용 UI** | **Oobabooga Text-Generation-WebUI** | 웹 브라우저에서 모델과 대화. 설정 조절 용이 |
| **양자화** | EXL2 / GGUF | 모델을 4비트로 압축하여 VRAM 절감 |

### Docker란? (간단히)

소프트웨어 실행 환경을 통째로 패키징한 것. RunPod에서 Pod을 생성할 때 Docker 이미지를 지정하면, 그 안에 CUDA, Python, PyTorch 등이 미리 설치되어 있어 별도 환경 설정이 필요 없습니다.

## 5. 파인튜닝 전략 (Fine-Tuning Roadmap)

### 사용 기법: QLoRA

- **Q** = Quantized (4비트 양자화), **LoRA** = Low-Rank Adaptation
- 24GB VRAM의 RTX 4090에서도 70B 모델 파인튜닝 가능
- 원본 모델 가중치는 건드리지 않으므로 실패해도 원본 복구가 쉬움

### 사용 도구: Unsloth

- 2026년 기준 가장 활발하게 업데이트되는 QLoRA 학습 도구
- HuggingFace Transformers에서 import 한 줄만 바꾸면 사용 가능
- 3x 빠른 학습 (2026 최신 packing 최적화)
- GGUF 포맷 직접 내보내기 지원

### 파인튜닝 프로세스 (단계별)

```
[1] 데이터 준비
    원본 데이터셋 다운로드
    → JSONL/ChatML 형식 변환
    → 품질 필터링, 중복 제거
    → 학습/검증 세트 분리

[2] 설정 파일 작성 (YAML)
    - 베이스 모델 경로
    - LoRA 하이퍼파라미터 (rank, alpha, target_modules)
    - 학습률, 배치 사이즈, 에폭 수
    - 데이터셋 경로 및 포맷

[3] 학습 실행
    python scripts/train.py --config configs/train.yaml
    → GPU에서 수 시간 실행
    → wandb로 loss 곡선 모니터링 (선택)

[4] LoRA Merge
    학습된 어댑터(adapter)를 원본 모델에 합침
    → 하나의 독립 모델 파일 생성

[5] 평가
    - 테스트 프롬프트 세트로 거절 여부 확인
    - 카테고리별 응답 품질 비교 (원본 vs 파인튜닝)
    - 필요 시 데이터/하이퍼파라미터 조정 후 재학습
```

### 하이퍼파라미터란?

학습을 제어하는 설정값들. 주요 항목:

| 파라미터 | 의미 | 일반적인 값 |
|---------|------|-----------|
| **learning_rate** | 한 번에 얼마나 크게 가중치를 수정할지 | 1e-4 ~ 2e-4 |
| **lora_rank (r)** | LoRA 어댑터의 크기. 클수록 표현력↑ but VRAM↑ | 16 ~ 64 |
| **lora_alpha** | LoRA 스케일링 계수. 보통 rank의 2배 | 32 ~ 128 |
| **epochs** | 전체 데이터를 몇 번 반복 학습할지 | 1 ~ 3 |
| **batch_size** | 한 번에 몇 개의 샘플을 학습할지 | 4 ~ 16 (VRAM에 따라) |
| **max_seq_length** | 입력 토큰 최대 길이 | 2048 ~ 4096 |

> 처음에는 Unsloth 기본값으로 시작하고, 결과를 보면서 조정하면 됩니다.

## 6. 추론 엔진 및 서빙 (Inference & Serving)

### 추론(Inference)이란?

학습된 모델에 입력을 넣고 출력을 받는 과정. "모델을 실행한다" = 추론한다.

### 엔진 선택

| 엔진 | 용도 | 장점 |
|------|------|------|
| **vLLM** | **프로덕션 서빙** (메인) | PagedAttention으로 최고 처리량. OpenAI-compatible API 기본 제공. 2026년 기준 Realtime API, Playground UI 추가 |
| **Oobabooga WebUI** | **실험/테스트** (보조) | 웹 브라우저에서 대화, 파라미터 실시간 조절, 다양한 모델 포맷 지원 |

### vLLM API 서빙 (간단 예시)

```bash
# vLLM 서버 시작
python -m vllm.entrypoints.openai.api_server \
    --model ./models/my-finetuned-model \
    --port 8000

# OpenAI 호환 API로 호출 가능
curl http://localhost:8000/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{"model": "my-model", "messages": [{"role": "user", "content": "Hello"}]}'
```

### Flash Attention이란?

Transformer 모델의 Attention 연산을 최적화하는 기법.
긴 컨텍스트(대화)에서 VRAM 소모를 크게 줄여줍니다.

- **적용 필수 상황**: 70B 모델 + 긴 대화 (4K+ 토큰)

## 8. 기술적 주의사항

### 가드레일 제거 관련

- **모델 레벨**: Abliteration/Dolphin으로 모델 자체의 거절 제거
- **엔진 레벨**: 추론 엔진(WebUI 등)에도 금지어 필터링이 있을 수 있음 → 설정에서 확인/해제 필요
- **시스템 프롬프트**: 모델 거절 제거 후에도, 시스템 프롬프트로 페르소나/행동 양식을 정의하는 것이 응답 품질에 큰 영향

### VRAM 관리

- 70B 모델 사용 시 대화가 길어지면 KV 캐시로 VRAM 급격 소모
- **Flash Attention** 적용 필수
- 컨텍스트 윈도우를 제한하거나 (예: 8K로 제한), 대화 요약 기법 활용

### 모델 버전 관리

- LoRA 어댑터를 여러 번 학습하면 버전이 누적됨
- **네이밍 컨벤션** 예시: `lora-dolphin-8b-v1`, `lora-dolphin-8b-v2-more-rp`
- wandb 등으로 실험 기록 추적 권장

### 보안/접근 제어

- Uncensored 모델 API를 외부에 노출하면 악용 가능성 존재
- 최소한 **API 키 인증** 적용 (vLLM의 `--api-key` 옵션)
- RunPod의 네트워크 설정에서 포트 접근 제한
