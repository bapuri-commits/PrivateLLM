## 1. 프로젝트 목표와 철학

### 최종 목표

**정말 사람과 대화하는 느낌**을 주는 AI 컴패니언.
단짝 친구, 연인과 같은 감정적 경험을 제공하며, 상용 AI의 불필요한 제약 없이 자유로운 대화가 가능하다.

### 왜 직접 만드는가?

현재 상용 LLM(ChatGPT, Gemini, Claude)은 혼자 사용하는 상황에서도 불필요한 제약이 많다:

| 제약 | 설명 | 이 프로젝트의 접근 |
|------|------|-----------------|
| **개인정보 기억 거부** | 이름, 취향 등을 기본 설정으로 기억시키려 하면 거절 | 장기 기억 시스템으로 자연스럽게 기억 |
| **역할 부여/RP 제한** | 특정 캐릭터 역할극을 시도하면 거부하거나 탈캐릭터 | 페르소나 시스템으로 일관된 캐릭터 유지 |
| **분위기 연출 제한** | 로맨틱/섹슈얼한 분위기를 선 넘지 않는 수준에서도 차단 | Abliterated 모델로 자연스러운 분위기 연출 |

### 핵심 철학

| 원칙 | 설명 |
|------|------|
| **품질 우선** | 속도/비용보다 대화 품질이 최우선. 큰 모델, 높은 양자화 |
| **사람 같은 경험** | 기억, 감정, 일관된 성격 — 기술이 아닌 경험 중심 설계 |
| **조립 + 확장** | 검증된 오픈소스 조합으로 시작, 부족한 부분만 강화 |
| **개인 전용** | 혼자 사용하는 환경. 외부 공개/서빙 목적이 아님 |

### "사람 같다"의 구성 요소

챗봇이 사람처럼 느껴지려면 단순히 좋은 모델만으로는 부족하다.
다음 4가지 요소가 모두 갖춰져야 한다:

```
[사람 같은 대화 경험]
├── 1. 대화 품질 ← 큰 모델 (27B~72B), 높은 양자화
├── 2. 기억 ← 장기 기억 시스템 (벡터DB + RAG)
├── 3. 인격 일관성 ← 페르소나/캐릭터 정의 시스템
└── 4. 몰입감 ← 채팅앱 같은 UI, 자연스러운 인터페이스
```

## 2. 핵심 개념 정리

### LLM (Large Language Model)

대규모 텍스트로 학습된 언어 모델. ChatGPT, Gemini의 핵심 엔진.
이 프로젝트에서는 오픈소스 LLM을 RunPod 클라우드 GPU에서 실행합니다.

### "Uncensored" / "Abliterated"

- **Censored 모델**: 상용 모델은 특정 주제에 "죄송합니다, 그건 답변할 수 없습니다" 식의 거절을 하도록 학습됨
- **Uncensored 모델**: 이 거절 메커니즘이 제거된 모델
- **Abliteration**: 모델 가중치에서 "거절 방향"을 수학적으로 찾아 제거하는 기법. 재학습 없이 변환 가능

### 양자화 (Quantization)

모델의 가중치를 더 적은 비트로 표현하여 메모리를 줄이는 기법.

- **FP16** (16비트): 원본. 품질 최고, VRAM 소모 최대
- **Q8** (8비트): 50% 절감. 품질 손실 거의 없음. **품질 우선 시 추천**
- **Q4** (4비트): 75% 절감. 약간의 품질 손실. 가성비 선택
- **GGUF**: llama.cpp 및 Ollama가 사용하는 양자화 포맷

### VRAM (Video RAM)

GPU 전용 메모리. 모델 가중치 + 연산 중간 결과가 여기에 올라감.
VRAM이 부족하면 모델이 로딩되지 않거나 크래시(OOM)가 발생.

### RAG (Retrieval-Augmented Generation)

모델이 직접 기억하지 못하는 정보를 **외부 데이터베이스에서 검색**하여 프롬프트에 주입하는 기법.
컴패니언 챗봇에서는 "과거 대화 내용, 사용자 정보"를 기억하는 핵심 메커니즘.

```
[사용자 입력] → [관련 기억 검색 (벡터DB)] → [기억 + 입력을 합쳐서 모델에 전달] → [응답]
```

### 벡터 DB (Vector Database)

텍스트를 숫자 벡터(임베딩)로 변환하여 저장하고, **의미가 비슷한 내용을 빠르게 검색**할 수 있는 DB.
"내 이름"을 저장하면, 나중에 "나에 대해 알려줘"라고 물었을 때 관련 정보를 찾아올 수 있다.

## 3. 하드웨어 & 인프라

### 로컬 환경 (참고용)

| 항목 | 스펙 |
|------|------|
| **GPU** | ASUS ROG STRIX LC RTX 3080 Ti (12GB GDDR6X) |
| **VRAM** | 12GB |
| **역할** | 가벼운 테스트, 개발 환경. 메인 추론은 RunPod |

### 메인 인프라: RunPod

품질 우선 전략 → 27B~72B 모델을 돌리려면 로컬 12GB VRAM으로는 부족.
**RunPod 클라우드 GPU**를 메인 추론 환경으로 사용한다.

#### RunPod 과금 모델

| 유형 | 과금 기준 | 특징 | 용도 |
|------|----------|------|------|
| **GPU Pods** | Pod가 **켜져 있는 전체 시간** (초 단위) | 세션 기반. 안 쓸 때 꺼야 함 | 개발, 테스트, 긴 세션 |
| **Serverless** | 요청 **처리 중인 시간만** (초 단위) | 순수 사용 시간. 대기 중 과금 없음 | 실사용, 간헐적 대화 |

- 개발/테스트 단계 → GPU Pods (모델 세팅, 프롬프트 실험)
- 실사용 단계 → Serverless 전환 검토 (대화할 때만 과금)

#### GPU별 비용 참고 (On-Demand 기준)

| GPU | VRAM | 시간당 비용 | 적합한 모델 |
|-----|------|-----------|-----------|
| **A5000** | 24GB | ~$0.36/hr | 27B Q4 |
| **A6000** | 48GB | ~$0.69/hr | 27B Q8, 72B Q4 |
| **A100** | 80GB | ~$1.49/hr | 72B Q8, 27B FP16 |
| **H100** | 80GB | ~$2.39/hr | 최고 성능 (속도 중심) |

> Spot 인스턴스를 쓰면 30~50% 할인 가능하지만 사용 중 회수 위험 있음. 긴 대화에는 On-Demand 추천.

## 8. Phase 4 — 장기 기억 시스템 (벡터DB + RAG)

> 목표: 사용자 정보와 과거 대화를 기억하고 활용하는 시스템 구축
> 비용: 개발 시간 (코드 작성)

### 장기 기억이 필요한 이유

LLM은 컨텍스트 윈도우(한 번에 처리할 수 있는 텍스트 길이) 내의 내용만 "기억"한다.
어제 대화, 지난주 대화, 사용자의 이름/취향 등은 **별도 시스템 없이는 기억 불가**.

### 기억 시스템 구조 (예상)

```
[사용자 입력]
    ↓
[기억 검색] ← 벡터DB에서 관련 기억 조회
    ↓
[프롬프트 구성] = 시스템 프롬프트 + 관련 기억 + 최근 대화 + 사용자 입력
    ↓
[모델 추론] → [응답]
    ↓
[기억 저장] ← 이번 대화의 핵심을 벡터DB에 저장
```

### 기억의 종류

| 종류 | 예시 | 저장 방식 |
|------|------|----------|
| **사용자 프로필** | 이름, 나이, 취향, 직업 | 명시적 key-value 저장 |
| **대화 요약** | "어제 학교에서 있었던 일 얘기" | 자동 요약 → 벡터 임베딩 |
| **감정 히스토리** | "최근 기분이 안 좋았음" | 감정 태그 + 타임스탬프 |
| **공유 경험** | "같이 본 영화, 나눈 농담" | 벡터 임베딩 |

### 후보 기술 스택

| 구성 요소 | 후보 | 비고 |
|----------|------|------|
| 벡터 DB | ChromaDB, Qdrant, Weaviate | ChromaDB가 가장 가볍고 시작하기 쉬움 |
| 임베딩 모델 | sentence-transformers, OpenAI Ada | 로컬 실행 가능한 오픈소스 추천 |
| 대화 요약 | LLM 자체로 요약 생성 | 메인 모델 또는 경량 모델 |
| 오케스트레이션 | LangChain, LlamaIndex, 직접 구현 | 추후 결정 |

## 9. Phase 5 — 채팅 UI 구축

> 목표: CLI가 아닌, 몰입감 있는 채팅 인터페이스 구축
> 비용: 개발 시간

### 왜 UI가 필요한가

- Ollama CLI로도 대화는 가능하지만, "사람과 대화하는 느낌"을 주기에는 한계
- 채팅앱 같은 UI가 있어야 몰입감이 생김
- 기억 시스템, 캐릭터 설정 등을 관리할 인터페이스도 필요

### 후보 기술

| 방식 | 기술 | 장점 | 단점 |
|------|------|------|------|
| 웹 앱 | Gradio, Streamlit | 빠른 프로토타이핑 | 디자인 제한 |
| 웹 앱 (커스텀) | React/Next.js + FastAPI | 완전한 자유도 | 개발 시간 |
| 데스크탑 앱 | Electron, Tauri | 네이티브 경험 | 개발 복잡도 |
| 기존 도구 활용 | Open WebUI, SillyTavern | 즉시 사용 가능 | 커스터마이징 한계 |

## 10. Phase 6+ — 확장 로드맵

Phase 5까지 완료된 이후, 필요에 따라 확장:

| 확장 | 설명 | 우선순위 |
|------|------|---------|
| **TTS (음성 합성)** | 텍스트 응답을 음성으로 변환. 대화 몰입감 극대화 | 높음 |
| **STT (음성 인식)** | 음성 입력 지원. 타이핑 대신 말로 대화 | 중간 |
| **아바타/프로필 이미지** | 캐릭터의 시각적 표현 | 중간 |
| **이미지 생성** | 대화 중 이미지를 생성/공유 (ComfyUI 연동) | 낮음 |
| **이미지 이해** | 사용자가 보낸 이미지를 이해하고 반응 (멀티모달) | 중간 |
| **API 서빙** | 다른 앱/서비스에서 호출 가능한 API | 낮음 |
| **파인튜닝** | 특정 대화 스타일로 모델 추가 학습 | 낮음 (Phase 4 결과에 따라) |

## 11. 기술적 주의사항

### 검열 제거 관련

- **모델 레벨**: Abliterated 모델은 거절 메커니즘이 이미 제거됨
- **도구 레벨**: Ollama 등 오픈소스 도구는 자체 검열이 없음
- **시스템 프롬프트**: 캐릭터의 행동 양식과 분위기 경계를 프롬프트로 제어

### RunPod 운용

- **Pod 사용 시**: 대화 끝나면 반드시 Pod 정지 (정지 중에는 디스크 비용만 소액 발생)
- **Serverless 전환 시**: 콜드 스타트(첫 요청 시 모델 로딩 대기)가 있을 수 있음
- **데이터 보안**: RunPod 서버에 개인 대화가 저장될 수 있으므로, 민감 데이터는 로컬 벡터DB에만 저장

### 모델/파일 관리

- 모델 파일은 수 GB~수십 GB 단위. 디스크 공간 확보 필요
- Ollama 모델 저장 경로: `C:\Users\[사용자]\.ollama\models\`
- 로컬 테스트용 모델과 RunPod 모델은 별도 관리

## 13. 부록: ML 학습 프로젝트 로드맵

> 이 프로젝트(PrivateLLM)와는 별개로, ML/AI 기초를 체계적으로 배우기 위한 독립 프로젝트 제안.
> 추후 파인튜닝이나 모델 커스터마이징이 필요할 때 이 역량이 활용됨.

### 추천 학습 경로 (단계별)

#### Step 1: 이론 기초 (1~2주)

| 리소스 | 내용 | 특징 |
|--------|------|------|
| **Andrej Karpathy - "Neural Networks: Zero to Hero"** (YouTube) | 신경망 기초부터 GPT까지 직접 코딩 | 전 OpenAI 연구원, 실습 중심, 무료 |
| **3Blue1Brown - "Neural Networks" 시리즈** (YouTube) | 직관적 시각화로 수학적 원리 설명 | 수학이 약해도 이해 가능, 무료 |

#### Step 2: 실습 — 작은 모델 직접 학습 (1~2주)

| 프로젝트 | 내용 |
|---------|------|
| **텍스트 분류기 만들기** | 영화 리뷰 감정 분석 (긍정/부정). HuggingFace Transformers 입문 |
| **작은 LLM 파인튜닝** | GPT-2 (124M 파라미터)를 특정 스타일로 파인튜닝. 전체 과정 체험 |

#### Step 3: 본격 LLM 파인튜닝 (2~4주)

| 프로젝트 | 내용 |
|---------|------|
| **QLoRA로 8B 모델 파인튜닝** | Unsloth + 커스텀 데이터셋으로 Llama 8B를 직접 학습 |
| **캐릭터 대화 데이터 학습** | 특정 대화 스타일을 학습시킨 어댑터 제작 |

#### Step 4: 심화 (선택)

- DPO (Direct Preference Optimization)
- RLHF (Reinforcement Learning from Human Feedback)
- 모델 아키텍처 이해 (Attention, Transformer)
